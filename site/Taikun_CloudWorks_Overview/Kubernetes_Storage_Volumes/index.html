    Storage Classes

    This document describes the concept of a StorageClass in Kubernetes. Familiarity withÂ volumesÂ andÂ persistent volumesÂ is suggested.

    A StorageClass provides a way for administrators to describe theÂ classesÂ of storage they offer. Different classes might map to quality-of-service levels, or to backup policies, or to arbitrary policies determined by the cluster administrators. Kubernetes itself is unopinionated about what classes represent.

    The Kubernetes concept of a storage class is similar to â€œprofilesâ€ in some other storage system designs.

    StorageClass objects

    Each StorageClass contains the fieldsÂ provisioner,Â parameters, andÂ reclaimPolicy, which are used when a PersistentVolume belonging to the class needs to be dynamically provisioned to satisfy a PersistentVolumeClaim (PVC).

    The name of a StorageClass object is significant, and is how users can request a particular class. Administrators set the name and other parameters of a class when first creating StorageClass objects.

    As an administrator, you can specify a default StorageClass that applies to any PVCs that donâ€™t request a specific class. For more details, see theÂ PersistentVolumeClaim concept.

    Default StorageClass

    You can mark a StorageClass as the default for your cluster. For instructions on setting the default StorageClass, seeÂ Change the default StorageClass.

    When a PVC does not specify aÂ storageClassName, the default StorageClass is used.

    If you set theÂ storageclass.kubernetes.io/is-default-classÂ annotation to true on more than one StorageClass in your cluster, and you then create a PersistentVolumeClaim with noÂ storageClassNameÂ set, Kubernetes uses the most recently created default StorageClass.
